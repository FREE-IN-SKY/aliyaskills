# Read Code Propose Change

## 目标

只读代码和配置，给出基于证据的改法。不要自动改文件，不要创建计划文档，不要提交。

## 工作流

1. 明确用户要实现的功能表现：用户点击什么、页面显示什么、数据如何变化、失败时看到什么。
2. 按当前仓库规则定位代码。若仓库有项目地图或 `AGENTS.md` 查找顺序，先遵守这些规则，再打开候选文件。
3. 只读相关入口、调用链、数据来源、配置和测试。最多先读 10 个候选文件；需要继续读时说明原因。
4. 用证据描述当前现状：哪个页面入口、哪个接口、哪个保存或读取路径已经存在，哪里断开。
5. 给出最小改法：改哪些文件、每处改什么、为什么改这里、牺牲什么。
6. 指出最脆弱点：需求变化时最先坏的位置，以及方案如何减少重复、保持职责清楚。
7. 给验证方式：用户能看到的结果、需要跑的现有测试或手动检查步骤。
8. 结尾明确写出：`本轮未修改文件。`

## 禁止事项

- 不编辑源码、测试、配置、文档、脚本、锁文件或生成文件。
- 不运行格式化、代码生成、迁移生成、安装依赖。
- 不创建 `PLAN.md`、`SUMMARY.md` 或其他工作流产物。
- 不执行会改 Git 历史或远端状态的命令。
- 不把猜测当结论；证据不足时说明缺少哪条证据。

## 输出格式

用短段落或项目符号输出：

- **我理解的需求**：用功能表现复述。
- **当前代码现状**：列文件和行号，说明用户实际会看到什么。
- **建议改法**：按文件列最小改动。
- **为什么这样改**：说明职责、复用、取舍。
- **风险点**：指出最容易坏的功能表现。
- **验证**：列用户可见检查和现有测试。
- **本轮未修改文件。**

## 升级规则

如果发现需求超过轻量分析范围，停止扩展并建议换更重流程：

- 需要定位真实 bug 根因：建议使用 `systematic-debugging-custom`。
